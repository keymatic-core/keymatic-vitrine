## SUMÁRIO EXECUTIVO

A Keymatic nasce da solidez de 20 anos da VTS Informática com uma proposta clara: traduzir décadas de experiência em soluções tecnológicas modernas para microempresários. Este documento define o escopo completo do site vitrine que atuará como **máquina de conversão** — transformando leads frios em clientes recorrentes de SaaS e Consultoria.

**Premissa Central:** O site não é um folder digital. É um **sistema de aquisição** que suporta confiança, avaliação e conversão antes de qualquer conversa comercial.

### 1.1 Diagnóstico do Cenário Atual (2026)

Com base na pesquisa de mercado realizada, os sites de tecnologia de alta conversão em 2026 compartilham estas características:

| Fator | Dado de Mercado | Impacto |
|-------|----------------|---------|

### 2.2 Detalhamento por Seção

#### 01. Navigation Bar
- **Função:** Orientação e acesso rápido
- **Design:** Fixa no topo, glassmorphism com backdrop-blur, colapsa em hamburger no mobile
- **CTA fixo:** Botão "Falar com Especialista" (WhatsApp) sempre visível
- **Comportamento:** Troca de opacidade ao fazer scroll (transparente → glass)

#### 02. Hero Section
- **Função:** Captura de atenção em 3 segundos
- **Headline sugerida:** *"Tecnologia que funciona enquanto você dorme."*
- **Subhead:** *"Da solidez de 20 anos da VTS nasce a Keymatic — e-commerce de elite, automação inteligente e consultoria em IA para quem quer crescer de verdade."*
- **CTAs duplos:** "Quero um diagnóstico gratuito" (primário) | "Conhecer soluções" (secundário, scroll suave)
- **Background:** Partículas sutis ou mesh gradient animado em tons escuros
- **Animação:** Texto revela progressivo (typewriter ou fade-up)

#### 03. Trust Bar (Barra de Confiança)
- **Função:** Eliminar ceticismo imediato
- **Conteúdo:** Logo de clientes atendidos | "+200 projetos entregues" | "+20 anos de mercado" | "99.9% uptime"
- **Design:** Carousel horizontal suave, monocromático (logos em branco sobre dark)
- **Critério:** Só incluir números verificáveis

#### 04. Os 3 Caminhos (Showcase de Soluções)
- **Função:** Identificação rápida da dor do cliente
- **Layout:** 3 cards interativos com Tilt effect

**Card 1 — E-commerce de Elite:**
- Ícone: Carrinho + foguete
- Dor: *"Sua loja cai em dia de lançamento?"*
- Solução: Checkout integrado, gestão de Drops, estoque inteligente
- CTA: "Quero uma loja que aguenta o tranco"

**Card 2 — Automação Inteligente:**
- Ícone: Engrenagem + WhatsApp
- Dor: *"Quanto você perde por não responder às 3h da manhã?"*
- Solução: Recuperação de vendas, atendimento 24h, Evolution API
- CTA: "Quero automatizar meu atendimento"

**Card 3 — Consultoria em IA:**
- Ícone: Cérebro + relógio
- Dor: *"Sua equipe gasta 6h no que a IA faz em 6 minutos?"*
- Solução: Treinamento, implementação, ROI mensurável
- CTA: "Quero um diagnóstico de IA"

**Interação:** Hover → borda neon dinâmica | Click → expande com mais detalhes (accordion ou modal)

#### 05. Como Funciona
- **Função:** Reduzir a ansiedade do "e agora?"
- **Layout:** Timeline horizontal (desktop) / vertical (mobile) com 4 etapas
- **Etapas:** Diagnóstico Gratuito → Proposta Personalizada → Execução Ágil → Acompanhamento Contínuo
- **Animação:** Reveal on scroll, cada etapa aparece sequencialmente

#### 06. Cases e Resultados
- **Função:** Prova concreta de capacidade
- **Conteúdo:** 2-3 cases com métricas reais (antes/depois)
- **Formato:** Card com screenshot do projeto + métricas em destaque
- **Exemplo de métrica:** "Loja X: de 2s para 0.8s de carregamento = +34% de conversão"
- **Vídeo-depoimento:** Embed leve (lazy-loaded) de cliente real
- **Importante:** Métricas devem ser reais e verificáveis

#### 07. Diferenciais Técnicos
- **Função:** Converter o lead técnico/decisor que pesquisa infraestrutura
- **Design:** Estilo "painel de controle" com tipografia monoespaçada
- **Elementos:**
  - Stack visual: Django 6.0 | PostgreSQL | Redis | Celery
  - Selo LGPD com link para política de privacidade
  - Badge 2FA: "Seus dados com dupla camada de proteção"
  - Integração Melhor Envio: "Logística automatizada em 1 clique"
  - Painel `/healthz/` estilizado: indicadores de saúde do sistema (uptime, latência, status)
- **Efeito:** Números animados (count-up) ao entrar na viewport

#### 08. Prova Social Dinâmica
- **Função:** Reforço contínuo de credibilidade
- **Conteúdo:** Carousel de depoimentos com foto, nome, empresa e rating
- **Diferencial:** Live activity feed simulado — "João acabou de solicitar um diagnóstico" (se houver volume real)
- **Nota de cautela:** Activity feed só deve ser implementado com dados reais, nunca simulados

#### 09. Consultoria de IA — Porta de Entrada
- **Função:** Principal vetor de conversão (baixo atrito, alto valor percebido)
- **Headline:** *"Descubra quanto tempo sua equipe pode economizar com IA"*

  - Input: "Quantos funcionários?" + "Horas/dia em tarefas repetitivas?"
  - Output: "Sua empresa pode economizar X horas/mês com automação inteligente"

## PARTE III — ANÁLISE TÉCNICA: DJANGO vs NEXT.JS

### 3.1 Estudo Comparativo para o Contexto Keymatic

| Critério | Django 6.0 | Next.js 16 | Vencedor para Keymatic |
|----------|-----------|------------|----------------------|
| **Performance SSG** | Possível com django-distill, mas não é nativo | SSG/ISR nativo, Turbopack, Edge Runtime | **Next.js** |
| **Core Web Vitals** | Requer otimização manual pesada | Otimizado nativamente (Image, Font, Script) | **Next.js** |
| **Meta < 1.5s** | Difícil de atingir com templates Django | Atingível com SSG + CDN (Vercel) | **Next.js** |
| **Ecossistema SaaS Backend** | Admin, ORM, Auth, Celery nativos | Precisa de bibliotecas externas | **Django** |
| **Integração Evolution API** | Django REST Framework maduro | API Routes funcionam bem | **Empate** |
| **SEO** | Bom com templates server-side | Excelente com SSG + metadata API | **Next.js** |
| **Animações/Micro-interações** | Limitado (precisa de JS separado) | Framer Motion nativo no ecossistema | **Next.js** |
| **Glassmorphism + Dark Mode** | CSS manual | Tailwind CSS v4 + CSS variables | **Next.js** |
| **Custo de Deploy** | VPS + Nginx + Gunicorn | Vercel free tier (hobby) ou VPS | **Empate** |
| **Time-to-market** | Mais lento para frontend rico | Mais rápido para sites interativos | **Next.js** |

### 3.2 Recomendação Arquitetural

**RECOMENDAÇÃO: Arquitetura Híbrida (Opção Ideal)**

```
┌──────────────────────────────────────────────────┐
│                  FRONTEND                         │
│           Next.js 16 + React 19                   │
│     Tailwind CSS v4 + Framer Motion               │
│        Deploy: Vercel (Edge Network)              │
│                                                   │
│  ┌─────────────┐ ┌─────────────┐ ┌─────────────┐ │
│  │ SSG Pages   │ │ ISR Pages   │ │ API Routes  │ │
│  │ (Home,About)│ │ (Cases,Blog)│ │ (Forms,Lead)│ │
│  └─────────────┘ └─────────────┘ └─────────────┘ │
└───────────────────────┬──────────────────────────┘
                        │ HTTPS/REST
┌───────────────────────┴──────────────────────────┐
│                  BACKEND                          │
│        Django 6.0 + DRF (API only)                │
│     PostgreSQL + Redis + Celery                   │
│        Deploy: VPS ou Railway                     │
│                                                   │
│  ┌─────────────┐ ┌─────────────┐ ┌─────────────┐ │
│  │ CRM/Leads   │ │ Evolution   │ │ Admin Panel │ │
│  │ Management  │ │ API Bridge  │ │ (Django)    │ │
│  └─────────────┘ └─────────────┘ └─────────────┘ │
└──────────────────────────────────────────────────┘
```

**Justificativa:**

1. **Frontend Next.js** para o site vitrine: SSG garante carregamento < 1s, Framer Motion habilita todas as micro-interações desejadas, Tailwind v4 simplifica o design system Dark Glassmorphism
2. **Backend Django** para a lógica de negócio: aproveita a stack existente da VTS (Postgres/Redis), Django Admin para gestão de leads/CRM, Celery para processamento assíncrono (emails, webhooks)
3. **Separação de concerns:** o site vitrine é estático (muda raramente), o backend é dinâmico (processa leads, integra WhatsApp)

**Alternativa B — Django Monolítico (se a equipe não dominar React/Next.js):**

Se a equipe da VTS é exclusivamente Python/Django, é possível construir o site com:
- Django 6.0 + HTMX + Alpine.js para interatividade
- Tailwind CSS v4 para styling
- django-compressor + WhiteNoise para assets estáticos
- **Trade-off:** Animações mais limitadas, performance inferior ao SSG, mas mantém a stack unificada

### 3.3 Por que NÃO usar Django puro para o Vitrine

| Problema | Impacto |
|----------|---------|
| Templates Django geram HTML server-side a cada request | Latência adicional vs SSG pré-renderizado |
| Sem otimização nativa de imagens | Imagens pesadas = LCP ruim |
| Glassmorphism + Parallax exige JS complexo | Precisa de framework JS de qualquer forma |
| Deploy requer servidor always-on | Custo maior que CDN estático |
| Sem ISR nativo | Conteúdo novo exige deploy completo |

**Conclusão:** Django é excelente para o backend/CRM/API, mas o frontend vitrine se beneficia enormemente do Next.js.

### 4.2 Micro-interações Especificadas

| Elemento | Interação | Implementação | Performance |
|----------|-----------|---------------|-------------|
| **Cards dos 3 Caminhos** | Tilt 3D no hover | `react-tilt` ou CSS transform | GPU-accelerated |
| **Bordas Neon** | Gradiente animado na borda | CSS `@property` + animation | ~0ms layout shift |
| **Botões CTA** | Glow pulsante | CSS keyframes com box-shadow | Sem reflow |
| **Números** | Count-up ao entrar na viewport | `Intersection Observer` + requestAnimationFrame | Lazy |
| **Seções** | Reveal on Scroll | Framer Motion `whileInView` | Threshold-based |
| **Parallax** | Background layers com velocidade diferente | CSS `transform: translateZ()` | GPU only |
| **Navigation** | Glass → solid on scroll | `Intersection Observer` + CSS transition | Sem JS-per-frame |
| **Formulário** | Progresso visual (steps) | State machine com transições | Minimal rerender |

### 4.3 Regras de Performance para Efeitos Visuais

1. **Toda animação deve usar `transform` ou `opacity`** — nunca animar `width`, `height`, `top`, `left`
2. **`backdrop-filter` limitado a no máximo 3 elementos simultâneos na viewport** — blur é caro na GPU
3. **Parallax desativado em dispositivos com `prefers-reduced-motion`**
4. **Partículas/mesh no hero limitadas a 30fps e desativadas em mobile**
5. **Lazy loading para toda interação abaixo do fold**

## PARTE V — PERFORMANCE E SEGURANÇA

### 5.1 Metas de Core Web Vitals

| Métrica | Meta | Estratégia |
|---------|------|------------|
| **LCP** (Largest Contentful Paint) | < 1.2s | SSG + CDN + next/image com priority |
| **FID/INP** (Interaction to Next Paint) | < 100ms | Minimal JS no critical path, code splitting |
| **CLS** (Cumulative Layout Shift) | < 0.05 | Dimensões explícitas em imagens/fonts, font-display: swap |
| **TTFB** (Time to First Byte) | < 200ms | Edge deployment (Vercel Edge Network) |
| **Total Bundle** | < 150KB gzipped (initial) | Tree-shaking, dynamic imports, no heavy libs |

### 5.2 Estratégia de Performance

```
CAMADA 1 — Build Time (SSG)
├── Páginas pré-renderizadas em HTML estático
├── CSS critical inlined no <head>
├── Fonts preloaded (Inter + JetBrains Mono)
└── Imagens otimizadas em WebP/AVIF com srcset

CAMADA 2 — Edge (CDN)
├── Cache headers agressivos para assets estáticos
├── Brotli compression
├── HTTP/3 quando disponível
└── Geo-distributed (Vercel Edge = 30+ PoPs)

CAMADA 3 — Runtime (Client)
├── Code splitting por rota
├── Lazy loading de seções abaixo do fold
├── Intersection Observer para animações
├── Service Worker para cache offline (opcional)
└── Prefetch de rotas no hover de links
```

### 5.3 Segurança

#### Camada Frontend (Next.js)

| Proteção | Implementação |
|----------|---------------|
| **CSP** (Content Security Policy) | Headers via `next.config.js`, whitelist strict |
| **XSS** | React escapa por padrão, CSP bloqueia inline scripts |
| **CSRF** | Tokens automáticos nos formulários |
| **HTTPS** | Forçado via Vercel (HSTS incluído) |
| **Subresource Integrity** | SRI para CDN scripts externos |
| **Rate Limiting** | Middleware Edge para API routes de formulário |

#### Camada Backend (Django)

| Proteção | Implementação |
|----------|---------------|
| **CORS** | django-cors-headers com whitelist do domínio Next.js |
| **Authentication** | JWT com refresh token rotation |
| **2FA** | django-otp para acesso ao admin |
| **SQL Injection** | ORM Django (parametrized queries nativo) |
| **LGPD** | Consentimento granular, direito ao esquecimento, exportação de dados |
| **Secrets** | Environment variables, nunca no código |
| **WAF** | Cloudflare ou AWS WAF na frente do backend |
| **Backup** | PostgreSQL: backup diário automatizado, retenção 30 dias |

#### Conformidade LGPD — Checklist

- [ ] Banner de cookies com opção de recusa granular
- [ ] Política de privacidade acessível em todas as páginas
- [ ] Formulário de solicitação de dados pessoais (acesso/exclusão)
- [ ] Registro de consentimento com timestamp
- [ ] Dados armazenados em território brasileiro (ou com cláusula contratual adequada)
- [ ] DPO (Encarregado de Dados) identificado na política
- [ ] Logs de acesso a dados pessoais
